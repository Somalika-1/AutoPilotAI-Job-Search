## Checkpoint 9 — Job-board clients: RemoteOK + Arbeitnow

**Status: done.**

### What this checkpoint is, in plain words

Before this, the app could only score a resume against a job description you typed in yourself. This checkpoint adds a way to *search for real, live job postings* from two job boards (RemoteOK and Arbeitnow) without needing any signup or API key for either one. You send a search word like `"python"`, and the app goes and asks both job boards for their current listings, picks out the ones that mention your word, and hands you back one combined list.

Nothing is saved to the database yet — a search just fetches fresh results each time and returns them. Saving a specific job for later is a separate, later checkpoint (11).

### Questions asked & decisions

**Q: Which two job boards go first?**
Decision: RemoteOK and Arbeitnow.
Why: both publish their job listings as a plain public JSON feed that anyone can call — no signing up, no API key, no login. That makes them the easiest possible starting point, exactly like ARCHITECTURE.md planned. Adzuna and USAJobs (Checkpoint 10) both need a free API key, which is one extra setup step, so they come after.

**Q: Neither job board lets you search by keyword through their API — what do we do?**
Decision: fetch each board's current list of jobs (RemoteOK returns about 100 recent postings, Arbeitnow similar), then filter it ourselves in Python — check if the search word appears in the job's title, company name, description, or tags.
Why: it's honest and simple. It was tempting to assume the API had a `?search=` parameter, but testing it directly (see "How" below) showed it didn't reliably change the results. Filtering after fetching guarantees correct behavior no matter what the provider actually supports.

**Q: What happens if one job board is slow or down when a user searches?**
Decision: skip that one board and still return results from the other, rather than failing the whole search.
Why: a user searching jobs shouldn't get an error just because one of two providers had a hiccup — partial results are much more useful than none.

**Q: Does this endpoint need login, like everything else in the app?**
Decision: yes, `GET /jobs/search` requires a valid JWT, same as every other route except `/health` and `/auth/*`.
Why: consistency — there's no reason to make this one route public when the rest of the app's rule is "everything behind login."

### What was built

- **`backend/app/services/job_scraper/schemas.py`** — one shared shape for a job listing, `JobListing`: id, source ("remoteok" or "arbeitnow"), title, company, location, link, when it was posted, and a description. Both job boards' very different raw data gets converted into this same shape, so the rest of the app never has to care which board a job came from.
- **`backend/app/services/job_scraper/remoteok.py`** — calls `https://remoteok.com/api`, skips the one non-job item RemoteOK always puts first (a legal notice), and converts the rest into `JobListing`s.
- **`backend/app/services/job_scraper/arbeitnow.py`** — calls `https://www.arbeitnow.com/api/job-board-api`, same idea.
- **`backend/app/services/job_scraper/__init__.py`** — `search_jobs(query)`, the one function everything else calls. It runs both providers above and combines whatever comes back, quietly skipping any provider that errors out.
- **`backend/app/routes/jobs.py`** — the actual endpoint, `GET /jobs/search?query=...`. Requires login, just like every other route. Returns the combined list, nothing written to the database.
- **`backend/tests/test_jobs.py`** — tests using a fake (mocked) set of results, so the test suite doesn't depend on the real internet or the real job boards being up.

### Why this shape

This is the exact same pattern already used for Gemini in `ai_engine`: one shared, predictable shape (`JobListing`, like `MatchResult`), with each real provider's messy format converted into it in its own small file. Adding a third provider later (Checkpoint 10) means writing one more small file with a `search()` function — nothing else in the app needs to change.

### How to run / verify it

```
cd backend && ./venv/Scripts/activate && uvicorn app.main:app --reload
```
Then, with a valid login token:
```
curl -H "Authorization: Bearer <token>" "http://localhost:8000/jobs/search?query=python"
```

**Verification actually performed:**
- `pytest -q` → 20 passed (17 from before + 3 new), using fake/mocked results — no real internet call happens during the automated test suite
- **Live-checked both real APIs directly** with `curl`, before writing any code, to see their actual field names (`position` vs `title`, `date` vs `created_at`, etc.) instead of guessing from memory or documentation that might be stale
- **Live-ran the real `search_jobs("python")` function** (not mocked) against both real job boards — it returned 10 real, current job postings mixing both sources (e.g. "Lead Data Scientist" at Brigit from RemoteOK, "Founding AI Engineer" from Arbeitnow), confirming the whole pipeline — fetch, filter, combine — actually works against the live internet, not just against fake test data
- Tried adding `?search=python` to Arbeitnow's URL directly to see if it changes the results — inconclusive (the live feed changes between requests anyway), which is exactly why client-side filtering was chosen instead of trusting an undocumented parameter

### Notes / gotchas

- RemoteOK's API has no `location`/`date_posted` filtering at all, and Arbeitnow's isn't confirmed to work — that's exactly what Checkpoint 10 needs to add "for real," likely via Adzuna/USAJobs (which do document filtering) rather than relying further on these two.
- No frontend page for this yet — that's Checkpoint 13, once saving/listing also exist. Right now this endpoint is only reachable via `curl`/Swagger UI (`/docs`), same honest gap V1 had between backend-only checkpoints and their eventual frontend wiring.
- RemoteOK occasionally returns a job with an empty `location` or missing salary fields — handled by treating empty strings as "unknown" (`None`) rather than showing a blank field.

## Checkpoint 10 — Add Adzuna + USAJobs, plus location/date filters

**Status: done, with one honest caveat — see "Notes / gotchas."**

### What this checkpoint is, in plain words

Checkpoint 9 could only search two job boards that need no signup at all. This checkpoint adds two more — Adzuna and USAJobs — which both need a free API key first. It also adds two new search options: filter by `location` (e.g. "Berlin") and by `date_posted` (e.g. only jobs from the last 7 days). Both filters now work across all four job boards, not just the new ones.

Importantly: if you haven't signed up for an Adzuna/USAJobs key yet, nothing breaks. Those two providers just quietly return no results, and your search still works fine with RemoteOK + Arbeitnow, exactly like before this checkpoint.

### Questions asked & decisions

**Q: Adzuna and USAJobs both need a signup + API key. What happens if the key is missing?**
Decision: that provider returns an empty list instead of raising an error, treated exactly the same as a provider being temporarily down.
Why: a user (or this environment, while building it) might not have signed up for both keys yet. The app should degrade gracefully — two working providers is strictly better than a search that errors out because two optional providers aren't configured.

**Q: Do RemoteOK and Arbeitnow (from Checkpoint 9) get the new `location`/`date_posted` filters too, or only the two new providers?**
Decision: all four providers now accept the same `search(query, location, date_posted)` shape.
Why: consistency — a user filtering by location shouldn't get unfiltered results from two of the four providers just because those two don't have a native filter API. For RemoteOK/Arbeitnow, the same "fetch everything, then filter in Python" approach from Checkpoint 9 was extended to also check location (substring match) and posting date (skip anything older than the requested window). Adzuna and USAJobs do support these as real API parameters, so their filtering happens on the provider's own server, which is more efficient.

**Q: How should "date_posted" be represented?**
Decision: four fixed choices — `24h`, `3d`, `7d`, `30d` — same as ROADMAP.md already specified, converted internally to "number of days" (1/3/7/30).
Why: a small fixed set is simple to validate (FastAPI rejects anything else with a 422) and maps directly onto every provider's own days-based filtering, instead of asking the caller for a raw date they'd have to compute themselves.

### What was built

- **`backend/app/config.py`** — four new optional settings: `ADZUNA_APP_ID`, `ADZUNA_APP_KEY`, `ADZUNA_COUNTRY` (defaults to `"us"`), `USAJOBS_API_KEY`, `USAJOBS_USER_AGENT`. All blank by default.
- **`backend/app/services/job_scraper/date_filters.py`** — turns `"24h"`/`"3d"`/`"7d"`/`"30d"` into either a day-count (for providers with a native "days old" parameter) or a cutoff date/time (for the two providers that need in-app filtering).
- **`backend/app/services/job_scraper/adzuna.py`** — calls Adzuna's search API with your `app_id`/`app_key`; returns `[]` immediately if either is blank.
- **`backend/app/services/job_scraper/usajobs.py`** — calls USAJobs' search API with your API key and a required `User-Agent` header (USAJobs asks that this be the email you registered with); returns `[]` immediately if not configured.
- **`backend/app/services/job_scraper/remoteok.py` / `arbeitnow.py`** — updated to also accept and apply `location`/`date_posted`, the same way `query` was already being filtered in Checkpoint 9.
- **`backend/app/routes/jobs.py`** — `GET /jobs/search` gained `location` (free text) and `date_posted` (must be one of the four allowed values, otherwise FastAPI returns `422` automatically) query parameters.
- **`.env.example` / `backend/.env`** — documented the four new settings, with links to each provider's free-signup page.
- **`backend/tests/test_job_scraper_providers.py`** (new) — tests for the date-filter conversion logic, plus tests for Adzuna/USAJobs that feed in a fake response shaped exactly like each provider's real documented JSON, checking it gets mapped to `JobListing` correctly.
- **`backend/tests/test_jobs.py`** — updated for the new `search_jobs` signature, plus two new tests checking `location`/`date_posted` are accepted and that a bad `date_posted` value is rejected.

### Why this shape

Same adapter pattern as Checkpoint 9 (and the same one `ai_engine` already uses for Gemini) — one shared `JobListing` shape, one small file per provider. Adding Adzuna/USAJobs didn't require touching anything about how RemoteOK/Arbeitnow work, only adding two more files and one more entry in the list `search_jobs()` loops over.

Treating "no key configured" the same as "provider is down" (return nothing, don't error) means this feature can ship and be reviewed *before* anyone has signed up for Adzuna/USAJobs credentials — exactly the situation this was actually built in.

### How to run / verify it

```
cd backend && ./venv/Scripts/activate && uvicorn app.main:app --reload
```
```
curl -H "Authorization: Bearer <token>" "http://localhost:8000/jobs/search?query=python&location=Berlin&date_posted=7d"
```
To turn on Adzuna/USAJobs for real: sign up free at https://developer.adzuna.com and https://developer.usajobs.gov, then fill in the four settings in `backend/.env` (see the comments added there) and restart the server — no code change needed.

**Verification actually performed:**
- `pytest -q` → 28 passed (20 from before + 8 new)
- **Live-ran the real `search_jobs()` function against RemoteOK + Arbeitnow** with the new filters, against the real internet: searching `"python"` with `location="Stuttgart"` correctly narrowed 10 unfiltered results down to exactly the 3 that are actually in Stuttgart — proof the location filter genuinely filters, not just accepts the parameter and ignores it
- Also live-checked that `location="remote"` legitimately returned 0 results for a `"python"` search where none of the current live listings happened to have the literal word "remote" in their location field — confirmed this is correct behavior (a strict, honest substring match), not a bug, by manually inspecting what location values the unfiltered results actually had
- Live-confirmed `adzuna.search()`/`usajobs.search()` both return `[]` with no error when called with no key configured (their intended "not set up yet" behavior)
- **Adzuna and USAJobs' actual search calls were not live-tested**, since no real API key for either was available in this environment (see "Notes" below) — instead, each adapter is unit-tested against a fake response built to match that provider's own published API documentation field-for-field (`id`/`title`/`company.display_name`/... for Adzuna, `MatchedObjectDescriptor.PositionTitle`/... for USAJobs)

### Notes / gotchas

- **Honest limitation, same shape as V1's browser-testing gap: Adzuna and USAJobs have not been called for real.** No API key for either was available while building this, so `adzuna.search()`/`usajobs.search()` are verified against each provider's own documented response shape (via mocked tests) but not against what the live API actually returns today. Documentation can drift from reality. **Recommend**: sign up for both free keys, add them to `backend/.env`, and run one real search (e.g. `curl .../jobs/search?query=python`) to confirm real results come back with the right fields filled in, before fully trusting this checkpoint.
- Adzuna requires a two-letter `country` code in the URL itself (not just a query parameter) — defaults to `"us"`; change `ADZUNA_COUNTRY` if you want a different country's listings.
- USAJobs specifically requires the `User-Agent` header to be the email address you registered with when signing up for the key — sending a made-up value is rejected by their API (confirmed by testing the real endpoint with no key at all: it returns a `401 Unauthorized`, which is how the header/key requirement was confirmed to exist, even without a working key to test the happy path).
- `date_posted` doesn't currently do anything visible when testing RemoteOK/Arbeitnow's live feed, since every listing in their current feed is well within the last 30 days anyway — the filter logic is verified correct by testing the conversion function directly (`cutoff_for("7d")` etc.), not by watching it visibly exclude an old RemoteOK/Arbeitnow listing in a live run.

## Checkpoint 11 — Save a search result

**Status: done.**

### What this checkpoint is, in plain words

Up to now, searching for jobs (Checkpoints 9-10) never remembered anything — every search asked the job boards fresh and threw the results away afterward. This checkpoint adds a "keep this one" button, in API form: `POST /jobs/save`. You send it one job listing (exactly what `GET /jobs/search` gave you), and it gets written into the database under your account. Trying to save the exact same listing twice is rejected, so your saved list can't fill up with duplicates.

### Questions asked & decisions

**Q: New table, or reuse the existing `job_descriptions` table?**
Decision: reuse `job_descriptions` (add four new columns to it), not a new table.
Why: `job_descriptions` already stores "a job description belonging to a user," which is exactly what a saved job is — it even already had a `source` column from Checkpoint 2 that was built anticipating this. A saved job is just a `job_descriptions` row with more fields filled in (`url`, `location`, `external_id`, `posted_at`) than a manually-pasted job description has. This also means a saved job can go straight into the existing match flow later (Checkpoint 13) with zero extra plumbing — it's already the same table `POST /matches` reads from.

**Q: How do we stop someone from saving the same listing twice?**
Decision: two layers — the endpoint checks first and returns a friendly `409 Conflict` if it's already saved, and the database itself also has a real uniqueness rule (`user_id` + `source` + `external_id` must be a new combination), as a backstop.
Why: the first layer gives a clean, expected error message. The second layer is the actual guarantee — even if two requests somehow raced each other, the database itself would refuse the second insert rather than silently allowing a duplicate. Two people can each save the same listing, though — the uniqueness only applies per-user, not globally, since two different users saving the same job is completely normal and not a duplicate at all.

**Q: Are the new columns required?**
Decision: no, all four (`url`, `location`, `external_id`, `posted_at`) are optional (nullable).
Why: this table already had rows from Checkpoint 5 (manually-pasted job descriptions), which have none of these fields and never will. Making them optional means the existing rows didn't need to change at all — this was a purely additive change to the table, nothing existing had to be touched or backfilled.

### What was built

- **`backend/app/models/job_description.py`** — four new columns (`url`, `location`, `external_id`, `posted_at`, all optional) plus the "can't save the same job twice" database rule described above.
- **A new Alembic migration** (`2e27e6298985_add_url_location_external_id_posted_at_.py`) — applied directly to the real database (the same Neon Postgres the deployed app uses), the same way every earlier schema change in this project has been.
- **`backend/app/schemas/job.py`** — `SavedJobOut`, the shape returned when a job is saved.
- **`backend/app/routes/jobs.py`** — `POST /jobs/save`. Requires login. Checks for a duplicate first (same user + same source + same external listing id); if none exists, writes the new row and returns it.
- **`backend/tests/test_jobs.py`** — new tests: saving works, saving the same listing twice is rejected, two different users can each save the same listing, and saving without logging in is rejected.

### Why this shape

Same reasoning already used everywhere else in this project: don't invent a new table/concept when an existing one already does 90% of the job (see also: reusing `job_descriptions.source`, planned back in Checkpoint 2, for exactly this V2 use case). The two-layer duplicate check (friendly error + real DB constraint) is the same "defense in depth" idea as the ownership checks already used on `/matches` — the API layer gives a nice message, the database layer is what actually can't be gotten around.

### How to run / verify it

```
cd backend && ./venv/Scripts/activate && uvicorn app.main:app --reload
```
```
curl -X POST -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
  -d '{"external_id":"1134476","source":"remoteok","title":"Lead Data Scientist","company":"Brigit","location":"Remote","url":"https://remoteok.com/remote-jobs/remote-lead-data-scientist-brigit-1134476","posted_at":"2026-07-07T20:14:56+00:00","description":"..."}' \
  "http://localhost:8000/jobs/save"
```

**Verification actually performed:**
- `alembic upgrade head` run for real against the live database (same one the deployed app uses) — confirmed clean, no errors, migration file reviewed by hand before applying (all four new columns nullable, so nothing about the existing data could break)
- `pytest -q` → 32 passed (28 from before + 4 new), including a real end-to-end duplicate-save test — this test only passes if the database's own uniqueness rule is actually in place and working, not just the in-app check, since both layers would need to agree for the test to be meaningful
- Confirmed via a dedicated test that two different users saving the identical listing both succeed (proving the uniqueness rule is correctly scoped per-user, not global)

### Notes / gotchas

- No `GET /jobs/saved` yet (Checkpoint 12) — right now there's no way to see your saved jobs except querying the database directly. Saving works, but "did it actually save?" can currently only be confirmed by the `201` response itself or a manual DB check, not by listing them back through the API.
- This migration was applied directly to the shared Neon database that the deployed (Render) backend also uses — same as every prior migration in this project. It's purely additive (new nullable columns + one new constraint), so existing data and the currently-deployed backend were unaffected by applying it.
