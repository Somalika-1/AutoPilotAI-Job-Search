# V1 — Implementation Log

Living doc for V1. Every checkpoint (see [ROADMAP.md](./ROADMAP.md) for the checklist) gets one section here, filled in **as it's built**, in this shape:

- **Questions asked & decisions** — anything that had more than one reasonable answer, what was chosen, and why
- **What was built** — the concrete files/features
- **Why** — the reasoning behind how it was built, not just what
- **How** — how to run/verify it
- **Notes / gotchas** — anything that cost time or is easy to get wrong again later

ARCHITECTURE.md is the *target* design; this file is the *actual* build record, checkpoint by checkpoint.

Written for a backend dev coming from Java/Spring Boot/MySQL, not from Python/JS — new tools get a plain-language explanation or a mapping to their closest Java/Spring/MySQL equivalent (see the table below) the first time they show up, instead of assumed knowledge.

## Rosetta stone (Java / Spring Boot / MySQL → this stack)

You already know backend systems — just not these specific tools. Same concepts, different names:

| You know | Here | Notes |
|---|---|---|
| Spring Boot app + embedded Tomcat | FastAPI + Uvicorn | `create_app()` in `main.py` is your `@SpringBootApplication` equivalent; Uvicorn is the server that runs it, like embedded Tomcat runs a Spring app |
| `@RestController` + `@GetMapping` | `APIRouter` + `@router.get(...)` | Same idea — a function decorated to handle one route, grouped into a router class |
| `application.properties` + `@ConfigurationProperties` | `.env` file + `pydantic-settings` `Settings` class | A typed config object that reads environment variables — same job |
| JPA / Hibernate | SQLAlchemy | The ORM. `Mapped[...]` + `mapped_column(...)` ≈ `@Entity` + `@Column`; `relationship(...)` ≈ `@OneToMany` / `@ManyToOne` |
| Flyway / Liquibase | Alembic | Generates versioned migration scripts from your ORM models instead of hand-writing SQL |
| MySQL | PostgreSQL | Same relational mental model. Postgres adds a native `JSONB` column type (queryable JSON in a column) — used here for LLM output like `missing_skills` |
| JDBC driver | `psycopg` | The actual driver SQLAlchemy talks through to reach Postgres |
| `pom.xml` / `build.gradle` | `requirements.txt` + a `venv` | Dependency list; `venv` is a project-local Python install so dependencies don't leak between projects |
| Thymeleaf/JSP (server renders HTML) | React (via Vite) | Now a separate app: the browser renders the UI and calls your API with `fetch`, instead of the server generating HTML pages |

## Checkpoint 0 — Repo & docs

### Questions asked & decisions

| Question | Decision | Why |
|---|---|---|
| Auth: hand-rolled JWT or Clerk? | Hand-rolled JWT (FastAPI + passlib + python-jose) | Stronger interview story ("I built auth") than wiring up a SaaS; keeps all user data in one Postgres DB instead of adding an external auth provider dependency |

| Python tooling: venv+pip or `uv`? | venv + pip + `requirements.txt` | Matches what FastAPI tutorials use; one less new tool to learn while also learning FastAPI, Postgres, and the OpenAI API for the first time |

### What was built

`README.md`, `ARCHITECTURE.md`, `ROADMAP.md`, `.gitignore`, `.env.example`; git repo initialized with one commit (`509ec59`). No application code — deliberately docs/scaffolding only.

### Why

The original project idea (see chat history / README's "Why this project") was solid but risked the two classic beginner failure modes: (1) too much infrastructure before any feature works, and (2) scraping LinkedIn/Indeed directly, which tends to get accounts/IPs banned before V2 is even usable. Both were designed out before any code was written, and the plan was reviewed with the user (via plan mode) before committing to it.

Beyond the four questions above, the plan also made these unprompted improvements to the original sketch, recorded here since they explain *why* later checkpoints are shaped the way they are:
- **Auth moved into V1** (not deferred to V2/V3) — the DB schema is much easier to get right from day one than to retrofit foreign keys onto `users` after V2/V3 add saved jobs and application tracking.
- **DB schema for all of V1–V3 sketched upfront** (in ARCHITECTURE.md: `users`, `resumes`, `job_descriptions`, `matches`, `applications`) so each checkpoint slots into one coherent schema instead of ad-hoc tables.
- **OpenAI structured outputs** (`response_format={"type": "json_schema", ...}`) planned instead of prompting "return JSON" as free text — materially more reliable.
- **`gpt-4o-mini` as the default model**, configurable via env var, for cost control while iterating.
- **Folder structure**: `resume_parser/`, `ai_engine/`, `job_scraper/` nested under `backend/app/services/` (not top-level siblings of `backend/`) so they're normal importable Python submodules of one package.
- **Dockerfile deferred** to the deployment checkpoint — not needed for local dev with `uvicorn`/`vite`.

## Checkpoint 1 — Backend + frontend skeletons

### Questions asked & decisions

Two more architectural questions came up after Checkpoint 0 was reviewed, before writing any Checkpoint 1 code:

**Q: Next.js or plain React for the frontend?**
Decision: switched from the originally-planned Next.js to **React + Vite + React Router**.
Why: Next.js's main selling points (server-rendered pages, its own API routes) exist to let a frontend act as its own backend. This project already has a real backend (FastAPI), so none of that gets used — the frontend is just a browser app calling your API with `fetch`, same as a Thymeleaf app would call a `@RestController`, except now it's a separate project instead of the same one. Plain React + Vite is the simpler, more direct match for that setup, with no unused framework machinery to learn on top of everything else.

*New concepts here with no direct Java/Spring equivalent, in plain terms:*
- **Vite** — the dev server + build tool for the frontend. Roughly: what `spring-boot:run` + hot reload is for your Spring app, but for the React/TypeScript code.
- **React** — a UI library where the page is built out of small reusable functions ("components") that return HTML-like markup. Replaces what a Thymeleaf/JSP template would render, except it runs in the browser and re-renders itself when data changes, instead of the server generating a new HTML page per request.
- **React Router** — decides which component to show for which URL, entirely in the browser (no server round-trip per page). Same *purpose* as `@RequestMapping`, different *place* it runs.
- **Tailwind CSS** — instead of writing CSS classes in a separate stylesheet, you style elements by combining small utility classes directly in the markup (e.g. `class="text-center gap-4"`).

**Q: Why Postgres and not MongoDB/SQLite/MySQL?** *(re-confirmed, no change)*
- vs **MongoDB**: the schema (`users → resumes → job_descriptions → matches → applications`) is genuinely relational — every table has foreign keys into another, and V3's analytics dashboard needs real joins/aggregations. Postgres's `jsonb` columns already give schema-flexible storage for LLM output (`missing_skills`, `strengths`) *inside* a relational table, so there's no upside to a document DB here, only the cost of hand-rolling referential integrity.
- vs **SQLite**: fine for local dev, but most PaaS platforms (Railway/Render) wipe the filesystem on redeploy, so a file-based DB doesn't survive in production, and it doesn't handle concurrent multi-user writes.
- vs **MySQL**: roughly comparable, but Postgres has stronger JSON support and is the more common default in FastAPI + SQLAlchemy + Alembic tutorials.

### What was built

**Backend** (`backend/`)
- FastAPI app factory in `app/main.py`, CORS enabled for `http://localhost:5173` (Vite's default dev port)
- `app/config.py` — `pydantic-settings` `Settings` class, reads `backend/.env` (mirrors the vars in root `.env.example`: `DATABASE_URL`, `JWT_*`, `OPENAI_*`). Schema only — not consumed by any route yet, that starts at Checkpoint 2/3.
- `GET /health` route (`app/routes/health.py`) returning `{"status": "ok"}` — the only real endpoint so far.
- `tests/test_health.py` — one pytest smoke test via `TestClient`.
- `requirements.txt`: fastapi, uvicorn[standard], pydantic-settings, pytest, httpx

**Frontend** (`frontend/`)
- Scaffolded with `npm create vite@latest frontend -- --template react-ts` (React 19, Vite 8, TypeScript)
- Tailwind CSS v4 via the `@tailwindcss/vite` plugin
- React Router (`react-router-dom`); `src/router.tsx` → currently just `/` → `src/pages/Home.tsx`
- `src/lib/api.ts` — fetch wrapper reading `VITE_API_BASE_URL` (default `http://localhost:8000`)
- `Home.tsx` calls the backend `/health` endpoint and renders live status ("checking" / "ok" / "unreachable")
- Removed default Vite template cruft (`App.tsx`, `App.css`, `assets/`)

### Why

The point of this checkpoint wasn't features — it was proving the two halves of the stack actually boot and talk to each other before anything real (DB, auth, AI) gets layered on. That's why the only "feature" is a health check rendered end-to-end through the frontend rather than, say, a static landing page: it's a genuine integration smoke test (CORS config, env-based API URL, fetch call, live UI state) rather than two independent hello-worlds.

Tailwind was set up via its v4 Vite plugin (not the v3-style `postcss.config.js` + `tailwind.config.js`) because that's the current documented path for Tailwind v4 — using the old approach would have been outdated the moment it was written.

## Checkpoint 2 — Postgres models + Alembic migrations

Done.

### Questions asked & decisions

**Q: How to run Postgres locally, since neither Postgres nor Docker is installed on this machine?**
Options considered: free-tier cloud Postgres (Neon/Supabase), native Windows Postgres install, or Docker just for a DB container.
Decision: **free-tier cloud Postgres (Neon)**.
Why: fastest path with no local install, and it doubles as the eventual deployment DB (ARCHITECTURE.md already planned on a managed Postgres for deployment) — one less thing to redo later.

**Getting the Neon connection string** (steps given to the user, to run themselves since it requires creating a personal account):
1. Go to neon.tech and sign up (GitHub login is fastest)
2. Create a project (e.g. name it `autopilotai`), any nearby region, default Postgres version
3. On the project dashboard's **Connection Details** panel, copy the connection string: `postgresql://<user>:<password>@<host>.neon.tech/<dbname>?sslmode=require`
4. Edit the scheme from `postgresql://` to `postgresql+psycopg://` (keep the rest, including `?sslmode=require`) — needed because this project uses the `psycopg` v3 driver, not the legacy `psycopg2` one
5. Paste the edited string into `backend/.env` as `DATABASE_URL` (already git-ignored)

Note: Neon's free tier auto-suspends its compute after inactivity and wakes on the next query with a ~1s cold start — expected, not a bug.

### What was built so far

Using the [Rosetta stone](#rosetta-stone-java--spring-boot--mysql--this-stack) mapping from the top of this doc: SQLAlchemy = your Hibernate/JPA, Alembic = your Flyway/Liquibase.

- `app/database.py` — SQLAlchemy `engine` + `SessionLocal` (≈ a JPA `EntityManagerFactory`/`SessionFactory` pair) and `Base` (the equivalent of Hibernate's entity registry — every model class below "registers" itself here)
- `app/models/user.py`, `resume.py`, `job_description.py`, `match.py` — one model per ARCHITECTURE.md's schema. Each is basically a JPA `@Entity` class: `Mapped[int] = mapped_column(primary_key=True)` ≈ `@Id`, `ForeignKey("users.id")` ≈ `@ManyToOne`/`@JoinColumn`, `relationship(back_populates=...)` ≈ the matching `@OneToMany` on the other side. `matches.missing_skills` / `strengths` use Postgres's `JSONB` column type (MySQL's `JSON` column, but natively queryable)
- `app/models/__init__.py` — imports all four models in one place purely so Alembic sees the complete schema when it scans for changes (if a model file is never imported anywhere, Alembic won't know it exists — same gotcha as forgetting to add an `@Entity` to your persistence unit)
- `alembic/` scaffolded via `alembic init alembic`; `alembic/env.py` edited to pull the DB URL from `app.config.settings` (the same config object the whole app uses) instead of a second hardcoded URL in `alembic.ini`
- Added `sqlalchemy`, `alembic`, `psycopg[binary]` to `requirements.txt` (`psycopg` is the JDBC-driver equivalent — the actual thing that talks to Postgres over the wire)

### Why

Models use SQLAlchemy 2.0's newer typed style (`Mapped[...]`) rather than its older, more dynamic style — it's the current documented approach and gives real type checking on fields, closer to how you'd expect a typed `@Entity` to behave.

Worth calling out since it's a common point of confusion coming from Hibernate: Alembic's `--autogenerate` is **not** like Hibernate's `ddl-auto=update`. It doesn't touch the database itself — it *compares* your model classes against the current DB schema and writes out a plain Python/SQL migration **file** for you to read and run explicitly (`alembic upgrade head`). Nothing changes in the database until that second step. That's why the migration was reviewed by hand before applying it (see below) — same spirit as reviewing a Flyway migration script before merging it, except Alembic drafts the first version for you.

### How

```
cd backend
# real DATABASE_URL already in .env (Neon, postgresql+psycopg:// scheme)
alembic revision --autogenerate -m "create users, resumes, job_descriptions, matches tables"
alembic upgrade head
```

**Verification actually performed:**
- Tested the live connection directly (`engine.connect()`) before generating anything → connected OK
- `alembic revision --autogenerate` → correctly detected all 4 new tables, the `ix_users_email` unique index, all 3 foreign keys, and both `JSONB` columns; migration reviewed by hand before applying (file: `alembic/versions/7f0833c7b729_create_users_resumes_job_descriptions_.py`)
- `alembic upgrade head` → applied cleanly
- `inspect(engine).get_table_names()` against the live Neon DB → `['alembic_version', 'job_descriptions', 'matches', 'resumes', 'users']`, confirming the migration actually landed, not just that it ran without error
- `pytest -q` → still 1 passed (no regressions)

### Notes / gotchas

- Installed `psycopg` (v3) instead of the older `psycopg2`. SQLAlchemy defaults a bare `postgresql://` URL to `psycopg2`, which isn't installed — hit a `ModuleNotFoundError: No module named 'psycopg2'` on first import. Fixed by using the `postgresql+psycopg://` scheme everywhere (root `.env.example` and `backend/.env`) to explicitly select the v3 driver.
- Verified models at least *import* and register correctly without a live DB (`Base.metadata.tables.keys()` → all four tables) — `create_engine()` doesn't connect eagerly, so this worked even before the real `DATABASE_URL` existed.
- Neon's connection string as copied from the dashboard uses `postgresql://` — remember to rewrite it to `postgresql+psycopg://` every time a fresh one is issued (e.g. if the DB is ever recreated).
